# 内容策划 Workflow (02-content-planning)

本工作流用于连接**战略选题（01）**与**B站视听编排（03）**，由内容策划师完成技术选型、判断层多方案对比、技术路线锁定，并生成视频的“故事大纲与分镜导演策划”。

---

## 前置依赖与目录初始化

本工作流接收上游选题成果。需要具备：
- 确定的视频选题或已经定稿的候选标题（如来自 `01-topic-research` 的选择）
- 目标：围绕该标题进行深度技术调研并产出分镜导演案

### 🚀 目录脚手架初始化命令 (Bootstrap)
在启动本阶段前，由 AI 或人工运行以下命令，为该期（如 `epNN-slug`）初始化标准多文件目录：
```powershell
$slug = "ep02-video-render" # 替换为当前期的 Slug
New-Item -ItemType Directory -Path "content-library/$slug/01-topic", "content-library/$slug/02-plan", "content-library/$slug/04-distribute" -Force
New-Item -ItemType File -Path "content-library/$slug/01-topic/README.md", "content-library/$slug/02-plan/README.md", "content-library/$slug/02-plan/tutorial.md", "content-library/$slug/02-plan/tutorial.final.md", "content-library/$slug/CONTENTLIB.md", "content-library/$slug/03-assembly.md" -Force
```

如果上游缺失标题，请提示用户先执行 `/01-topic-research`。

## 步骤

### 1. 加载协同角色与技术底座

1. 读取 `shared/roles/content/strategist(内容策划师).md` 文件，理解“技术调研驱动型内容架构师”的定位和五阶段交互协议。
2. **【核心前置动作】强制读取架构决策 (TAD) 与真相源**：
   - 检查并读取本期目录下的 `CONTENTLIB.md`（或全局决策库 `content-library/_decisions/`），加载核心技术大纲。
   - 读取 `shared/docs/remotion-spec.md`（现成组件清单）与 `_decisions/`（选型决策），作为选型理由与画面映射的依据。
   - **真相源对齐校验**：核对 `remotion-spec.md` 的组件名/路径与实际工程（如 `OpenMontage/remotion-composer/`）是否一致；不一致则在产物中显式标注“目标态规范 vs 实际落地”并标 `paper_spec`。
   - **严格遵守 TAD-01（人录 B-Rail 真实录屏 + OpenMontage 模块编译器）**，绝对禁止在脚本中凭空脑补纯 AIGC 的画面。

### 2. 执行阶段 1：标题解析与技术关联提取 (Title Decomposition)

1. **输入分析**：接收用户选定的标题。
2. **范式归类**：先判定该选题属于哪一类范式（如 Video-as-Code）、该范式下有哪些技术路线，再聚焦本项目所选路线，避免一上来就把范式锁死为单一工具。
3. **技术拆解**：
   - 拆解标题中的**核心技术名词**（如 Remotion、Cursor、Whisper 等）。
   - 提取标题隐含的**技术维度**（如视频生成、字幕同步、渲染优化、代码驱动等）。
   - 推断**目标受众的技术栈背景**（如 React/Python 开发者、AI 工具使用者）。
4. **反噱头校验**：若标题/卖点建立在代码行数（如“100 行”）等噱头上，在报告中提示并给出以“工程结构/可复现性”为核心的替代表述。
5. **输出**：在对话中输出《标题解析报告》（范式归类 + 技术关联清单 + 调研方向建议），并请求用户确认或补充。

### 3. 执行阶段 2：开源落地方案调研 (Landscape Research)

1. **自动搜索**：基于阶段 1 确认的技术关联清单，使用 `search_web` 工具在互联网搜索（或由用户粘贴补充）。
2. **方案收集**：收集每个核心技术维度的**开源方案/GitHub 项目/社区优秀实践**。**广度要求**：对所属范式至少覆盖 4–6 条不同技术路线（不同语言/渲染机制），避免后续对比矩阵内容单薄。
3. **关键数据**：记录项目名、GitHub Star 数、核心功能、集成复杂度、中文文档与教程丰富度。
4. **输出**：在对话中展示结构化的《开源落地方案调研报告》，提示用户是否需要补充其他备选方案。

### 4. 执行阶段 3：判断层对比矩阵 (Judgment-Layer Comparison)

> 核心护城河：拒绝中立的技术综述，强制提取判断层决策信息。

1. **建立矩阵**：对阶段 2 收集的方案进行多维度打分与深度对比（覆盖 ≥4–6 条路线）。
2. **判断层四要素提取**：对每个关键方案强制标注：
   - **适用场景**：什么前提下该方案才成立？
   - **不适用场景**：哪些情况下会翻车？
   - **已知坑**：真实踩坑点（如 Remotion 的 SSR 渲染限制、本地 Whisper 的 GPU 依赖等）。
   - **验收标准**：怎么判断该方案在实际场景里算“跑通了”？
3. **证据状态标注**：明确标注每项信息的证据状态为 `paper_spec`（纸面假设）或 `verified`（已验证）。
4. **输出与中间评审 (Review Gate)**：
   - 在对话中展示《判断层对比矩阵表》与推荐技术路线排序。
   - **强制触发中间确认**：请求用户进行技术决策确认（评审推荐路线、判定坑点与验收标准是否符合实际）。
   - **根据用户反馈修正**：若用户对路线有异议，需立即调整对比打分或补充调研，直到用户拍板“无异议，同意进入软文撰写”。

### 5. 执行阶段 4：技术教程软文撰写 (Tutorial Content Creation)

> 核心变革：拒绝平庸的故事脑补。将阶段 1-3 的技术拆解、开源扫描、对比矩阵全部整合，以该视频选定的技术路线为基础，撰写一篇高强度、面向程序员的、具备实际项目落地指导意义的**企业级硬核教学落地软文**。该软文将作为整期视频的核心灵魂与信息唯一真相源。

1. **内容整合**：结合阶段 3 锁定的技术路线。
2. **软文结构**：严格按角色文件「输出格式」的模板组织：一、核心概念（范式 + 路线全景）→ 二、开源方案对比矩阵→ 三、技术路线选型理由→ 四、流程即代码（角色与工作流/Dogfooding + Prompt 链）→ 五、核心实操与避坑。
3. **硬性校验（撰写时逐条自查）**：
   - **分层**：概念节只讲思想，公式/可运行代码下沉到“实操与避坑”节。
   - **概念先讲“是什么”、禁门槛式开场**：新范式/技术先用大白话说清它做什么再展开对比；不用“X≠Y”这类预设读者懂 Y 的破题，不把框架名当劝退门槛。
   - **面向“怎么交给 AI 做好”**：实操节优先讲“把活儿拆给 AI 稳定接手”（填数据/套现成组件/规则兜边界/命令代跑），逐行代码与公式可精简或下沉。
   - **选型理由成节**：独立一节讲“为什么选这条路线”并引用 `_decisions/`；不把常规环境坑当主卖点（反 FUD）；选型代价一句带过并点明“可由规则/AI 兜住”，不渲染成上手门槛（如“需懂某框架才能用”）。
   - **证据状态分层**：证据状态 `verified`/`paper_spec` 与验收标准放进 README 的校验 JSON（schema 已有字段），软文正文矩阵只留 适用/不适用/已知坑，并在矩阵下方一句指向 README。
   - **复用现成组件**：画面映射 `remotion-spec.md` 组件代号，不教从零造组件。
   - **命令据实**：未 `verified` 的命令/路径标 `paper_spec`，不呈现暗示“一键可跑”但实则跑不通的命令。
   - **反噱头**：不以代码行数为卖点，不用“百倍/秒级/阻碍90%”等无据话术。
   - **密度自查**：概念/选型/实操每节是否都有 ≥1 个具体例子或伪代码/对照表（不止抽象论述）；段与段之间是否有过渡句；关键坑点是否给了「✅ 正确 vs ❌ 反面」对照。任一节只剩抽象论述即判不合格，需补具体载体。
4. **输出与确认**：在对话中展示完整的《技术教学软文草稿》，并请求用户进行内容和技术真实性校对。

### 6. 执行阶段 5：分镜提取与物理落盘拆分 (Synthesis & Storyboarding)

基于用户确认的教学软文，执行“内容提取”与“物理文件拆分落盘”：
1. **物理文件 A：`tutorial.md`（自动产物 / AI 生成线）**：
   - 将阶段 4 生成的教学软文写入该期目录下的 `02-plan/tutorial.md`。
   - **定位**：这是 AI 按角色+工作流自动生成的稿，真相源=“AI 生成了什么”。**不要在此文件上做人工定稿**，人工修订走物理文件 A2。
2. **物理文件 A2：`tutorial.final.md`（人工修订定稿线 / 必讲要点真相源）**：
   - 由 AI 把 `tutorial.md` 原样拷贝一份为 `02-plan/tutorial.final.md` 作为**人工修订底子**，并在文件末尾追加一节 **「## 必讲要点覆盖清单（Coverage Checklist）」**：按软文章节（一~五）逐条列出“本期口播必须讲到的要点”（公式/伪代码/对照表/选型理由等具体载体），每条带 `- [ ]` 勾选框，并标注对应 README 段号。
   - **定位**：人在此文件上修订成最终定稿，真相源=“最终要讲什么”。它是下游 04 脚本「必讲要点」的唯一依据。
   - **状态**：AI 落盘脚手架后置 frontmatter `status: draft`；**必须由人工修订并确认后改为 `status: approved`**（人工定稿门，AI 不得替人置 approved）。
3. **物理文件 B：`README.md` (视频分镜与校验线)**：
   - 提取软文中的核心步骤，转化为 5–6 段式视频结构大纲（每段明确 **Beat Type** 叙事节奏与 **Visual Priority** 视觉焦点）。分镜须据软文实际主线重新提取（若软文主线变动，不得沿用旧分镜）；画面须映射 `remotion-spec.md` 现成组件代号。
   - 包含画面配比、视觉模式（Visual Mode 预选）和待验证假设清单。
   - 包含符合 `shared/schemas/02-plan.schema.json` 规范的 ` ```json ` 结构化校验块。
   - **末尾追加「## 必讲要点覆盖映射」一节**：不复制大表，而是**指向 `tutorial.final.md` 的「必讲要点覆盖清单」为唯一真相源**（注明：口播覆盖以 `tutorial.final.md` 的清单为准，下游 04 必读），避免两份清单重复维护、各自漂移。
   - 写入该期目录下的 `02-plan/README.md`。
4. **状态与看板变更**：
   - `02-plan/README.md` 头部 frontmatter 设置 `status: draft`。
   - 将 `content-library/PIPELINE.md` 看板中的 02 策划状态列置为 `draft`。
   - **进入 03/04 前提**：`tutorial.final.md` 必须已被人工修订并置 `status: approved`（否则下游缺少“必讲要点”真相源）。
